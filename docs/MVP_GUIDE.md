# Melody Bubble MVP 구현 기준

## 1. 산출물 해석

네 개의 원본 산출물은 서로 다른 상세도를 가집니다.

- `Information architecture.png`는 인증·온보딩부터 홈, 근처, 라운지, 인박스, 마이, 오프라인 교환까지 전체 제품 경로를 정의합니다.
- `wireframe.png`는 홈, 근처, 사용자 상세, 라운지 목록·상세, 인박스, 1:1 채팅, 마이의 핵심 사용 흐름을 정의합니다.
- `DB.png`는 인증, 실시간 위치·음악, 소셜·채팅, 라운지·행사, 오프라인 교환을 포함한 서버 전체 스키마입니다.
- `기획서 초안.md`는 STOMP over WebSocket, 개인 Queue, 그룹 Topic, Foreground Service, Nearby Connections를 포함한 완성 제품 방향입니다.

Android MVP는 이 전체 범위를 한 번에 서버까지 구현하는 결과물이 아닙니다. 현재 저장소에서는 와이어프레임의 핵심 화면과 실시간 인터랙션을 데모 데이터로 끝까지 시연할 수 있게 만들고, 실제 서버 연동 경계는 별도 계약으로 고정합니다. URL 설정과 endpoint 상수는 준비되어 있지만 네트워크 어댑터는 아직 연결하지 않았습니다.

## 2. MVP 목표

1. 익명 주변 음악 탐색의 가치가 첫 실행에서 이해되어야 합니다.
2. 홈 → 근처 → 사용자 상세 → 리액션·팔로우가 끊기지 않아야 합니다.
3. 라운지에서 추천곡·투표의 다중 사용자 실시간 갱신을 시연할 수 있어야 합니다.
4. 인박스 → 맞팔 대화방의 1:1 메시지 흐름을 시연할 수 있어야 합니다.
5. 실제 서버가 없어도 데모가 가능하고, 나중에 REST/STOMP 구현으로 교체할 수 있어야 합니다.
6. 정확한 거리·방향·좌표를 UI나 영속 저장소에 남기지 않아야 합니다.

## 3. 데모 모드

`API_BASE_URL`과 `STOMP_WS_URL`이 모두 비어 있는 구성이 데모 기준입니다.

- 주변 사용자, 음악, 라운지, 알림, 채팅의 시드 데이터를 사용합니다.
- 사용자의 선택, 리액션, 팔로우, 메시지, 투표는 즉시 화면 상태에 반영합니다.
- 공유가 활성화되면 약 3.5초마다 가상 버블 위치와 캠퍼스 라운지 인원이 변해 Delta를 시연합니다.
- 채팅은 `PENDING`에서 약 450ms 뒤 `SENT`로, 오프라인 교환 동기화는 약 500ms 뒤 `SYNCED`로 바뀌는 데모 ACK를 사용합니다.
- 네트워크 장애나 외부 계정 없이 발표 흐름을 반복할 수 있어야 합니다.
- 데모 버블의 위치와 근접 상태는 시각화용 가상 값이며 실제 위치를 뜻하지 않습니다.
- 데모 데이터에는 실제 이메일, 전화번호, 위치, 음악 계정 식별자를 넣지 않습니다.

현재 시드는 외부에서 가져오는 데이터가 아니라 앱의 `DemoCatalog`에 정의된 발표용 fixture입니다.

| 영역 | 현재 데모 데이터 |
|---|---|
| 내 상태 | `JH Melody`, 주변 별칭 `Lime`, Indie·R&B·Calm·Night, 현재 곡 `Blue Night` |
| 근처 | `Mint`, `Wave`, `Nova`, `Echo` 네 개 만료형 handle과 가상 배치·유사도·넓은 근접 상태 |
| 인기 음악 | `Blue Night`, `Late Drive`, `Indie Rain`의 익명 listener/reaction 집계 |
| 라운지 | 캠퍼스 LIVE 42명, 도서관 LIVE 18명, 축제 SCHEDULED; 캠퍼스 추천곡 세 장과 장르 투표 |
| 인박스 | 리액션·팔로우·시스템 알림과 `Mint` 맞팔 대화방 한 개 |
| 오프라인 교환 | 최초에는 비어 있음; `Clover` 또는 `Moon` 데모 교환 후 Room에 생성 |

`Blue Night`의 외부 URL은 fixture 메타데이터입니다. 사용자 상세의 곡 액션은 값이 검증된 HTTPS URL일 때만 외부 앱으로 열고, URL이 없으면 데모 현재 곡만 바꿉니다. 시작 시나 선택 시 음악 사업자 API를 호출해 데이터를 수집하지 않습니다.

## 4. 실제 연동 모드

실제 연동은 두 URL이 모두 설정되고 해당 계약을 제공하는 백엔드와 Android 네트워크 어댑터가 있을 때만 완성됩니다. 현재 `DemoMelodyRepository`는 두 값이 각각 `https://`, `wss://`로 시작하면 데이터 출처를 `DEMO FALLBACK`으로 표시할 뿐, URL을 완전히 검증하거나 네트워크 요청을 보내지 않습니다.

- 후속 REST 어댑터는 로그인·초기 조회·설정 변경·과거 기록 조회를 담당합니다.
- STOMP는 주변 사용자 Delta, 개인 알림·채팅, 라운지·지역 집계를 담당합니다.
- STOMP 연결은 사용자당 하나를 유지하고 필요한 Queue와 Topic을 함께 구독합니다.
- JWT는 앱 소스나 `local.properties`에 저장하지 않고 로그인 응답 후 Android 보안 저장소에 보관하는 방향을 사용합니다.
- 서버는 수신자별 공개 범위와 차단 관계를 적용한 결과만 앱으로 전송해야 합니다.

두 값 중 하나만 존재하거나 HTTPS·WSS가 아니면 설정은 미완성입니다. 현재는 데모로 안전하게 동작하며, 후속 실제 어댑터는 사용자가 이해할 수 있는 설정 오류 상태를 보여야 합니다.

## 5. 발표용 실행 시나리오

### 시나리오 A: 주변 음악으로 연결

1. 데모 모드로 앱을 실행합니다.
2. 홈에서 `주변 공유`를 사용자가 직접 시작합니다.
3. 숫자 거리 대신 익명 버블과 취향 유사도, 공개된 현재 음악을 확인합니다.
4. 근처 탭에서 버블을 선택해 사용자 상세로 이동합니다.
5. 정해진 음악 리액션을 보내고 팔로우 상태가 바뀌는 것을 확인합니다.
6. 인박스에 리액션·맞팔 알림이 반영되는 것을 확인합니다.
7. 맞팔 대화방에서 메시지를 보내 ACK 또는 전송 완료 상태를 확인합니다.

### 시나리오 B: 라운지 실시간 인터랙션

1. 라운지 목록에서 캠퍼스 라운지를 선택합니다.
2. 참여자 수, 분위기, 추천곡 카드를 확인합니다.
3. 장르·분위기 투표에 참여하거나, 입장 상태에서 현재 음악 카드를 보냅니다.
4. 투표 비율 또는 새 카드가 화면에 즉시 반영되는 것을 확인합니다.
5. 라운지에서 나가 데모 참여 인원과 입장 상태가 바뀌는 것을 확인합니다.

### 시나리오 C: 개인정보 경계

1. 홈과 근처 화면 어디에도 상대의 좌표·정확 거리·실제 방위가 없는지 확인합니다.
2. 마이 화면에서 주변 발견, 정해진 리액션 수신, 오프라인 교환 설정을 확인합니다.
3. 공유를 중지하면 화면이 `주변 공유 꺼짐`으로 바뀌고 위치 이력을 표시하지 않는지 확인합니다.
4. 로컬 저장 데이터에 좌표나 이동 경로가 없는지 확인합니다.

## 6. 이번 구현 범위

### 포함

- Jetpack Compose 기반 핵심 화면과 하단 탐색
- 홈의 공유 상태·추상 레이더·가까운 음악·주변 리스너·인기곡
- 근처의 가상 `displayPosition` Canvas, 유사도 `60%+`·`75%+`·`90%+` 필터, 선택 카드
- 사용자 상세의 공개 현재 곡·공통 장르, 정해진 리액션 4종, 팔로우, 차단·신고 진입점
- 라운지 목록·상세, 추천곡 카드, 집계 투표의 데모 인터랙션
- 인박스와 1:1 채팅 데모 흐름
- 마이 화면의 주변 발견·리액션·오프라인 교환 설정과 로컬 기록 진입점
- 실제/데모 데이터 소스 경계
- Room `offline_exchange_local`, `sync_outbox`를 통한 데모 오프라인 교환·동기화 상태
- 기기 내부에 필요한 최소 비민감 공유·현재 음악 폴백 상태
- 위치 권한 요청 뒤에만 시작되는 Foreground Service, 알림 권한이 있을 때 표시되는 중지 액션, 앱 복귀 시 공유 상태 재조정
- 마이 화면의 알림 접근 설정 진입점, transport 알림 기반 현재 음악 폴백, 수동 음악 선택

### 이번 저장소의 백엔드로 오해하면 안 되는 항목

- PostgreSQL/PostGIS 서버
- Spring Boot REST/STOMP 서버와 RabbitMQ Broker Relay
- 운영 사용자 인증과 JWT 발급 서버
- 서버 기반 유사도 계산, 차단·신고·rate limit 집행
- FCM 운영 발송
- 음악 사업자 검색·스트리밍 API
- URL을 실제 호출하는 Retrofit/STOMP 네트워크 어댑터

## 7. 후속 구현 범위

우선순위 순으로 다음을 연결합니다.

1. 인증·프로필·공개 범위 REST와 안전한 토큰 저장
2. Presence 시작·heartbeat·종료와 개인 주변 Delta Queue
3. 수신자별 공개 범위·차단 정책이 적용된 PostGIS 주변 검색
4. 음악 리액션, 팔로우·맞팔, 채팅 저장·ACK·중복 방지
5. 라운지 가입 권한과 Topic 기반 카드·리액션·투표
6. Foreground Service의 좌표를 이력 없이 서버 adapter로 전달하고 기기·OS별 수명주기 검증
7. 현재 NotificationListener·수동 폴백에 더해 MediaSession 직접 감지와 음악 앱별 정규화
8. Nearby Connections 1:1 카드 교환과 오프라인 토큰 동기화
9. FCM, 신고·차단 운영 도구, 접근성·오류·재연결 강화

장기 위치 기록, 정확 거리·방향 제공, 자유 텍스트 그룹 채팅, 사진·파일, 음성·영상, 음악 직접 스트리밍, 복잡한 LLM 추천은 MVP 후속에도 기본 포함되지 않습니다.

## 8. 완료 판단

- 빈 URL로 설치·실행되고 발표 시나리오 A와 B를 반복할 수 있습니다.
- 홈에서 `DEMO LIVE` 또는 `DEMO FALLBACK` 출처가 보이고, 오프라인 기록의 빈 상태가 구분됩니다.
- 데이터 출처가 데모인지 실제인지 사용자와 개발자가 혼동하지 않습니다.
- 주변 버블에 실제 방향과 정확 거리로 해석 가능한 정보가 없습니다.
- Room·SharedPreferences·로그에 원시 좌표나 이동 이력이 없습니다.
- `./gradlew test`와 `./gradlew assembleDebug`가 통과합니다.
