# **근거리 음악 소셜 네트워크 서비스 기획안**

## **1\. 프로젝트 개요**

### **1.1 프로젝트명**

**Melody Bubble**

### **1.2 서비스 소개**

Melody Bubble은 주변에 있는 사용자들이 현재 듣고 있는 음악과 음악 취향을 익명 버블 형태로 확인하고, 음악 리액션·팔로우·채팅·실시간 음악 라운지를 통해 관계를 형성하는 Android 기반 근거리 음악 소셜 네트워크 서비스다.

사용자는 얼굴 사진, 정확한 위치, 실제 방향 대신 다음 요소로 표현된다.

* 익명 버블  
* 프로필 색상  
* 음악 파형  
* 현재 재생 상태  
* 멜로디 별칭  
* 음악 취향 유사도  
* 장르 및 분위기 태그

서비스의 온라인 실시간 통신은 **STOMP over WebSocket**을 중심으로 구현한다. 사용자별 메시지는 개인 Queue로, 지역·행사·음악 라운지처럼 여러 사용자가 동시에 참여하는 통신은 Topic으로 전달한다.

### **1.3 지원 플랫폼**

* Android 전용  
* 프로젝트 착수 시점의 최신 안정 Android 버전 지원  
* 최신 Android 버전까지 업데이트 가능한 테스트 기기 기준  
* iOS 개발 제외  
* 태블릿·웨어러블 제외  
* 스마트폰 세로 화면 우선 지원

Android 빌드 설정은 프로젝트 착수 시점의 최신 안정 API를 기준으로 정한다.

compileSdk: 최신 안정 API

targetSdk: 최신 안정 API

minSdk: 테스트 기기 중 가장 낮은 API

### **1.4 서비스 운영 모드**

#### **온라인 모드**

* 사용자 인증

#### **오프라인 근거리 교환 모드**

## **2\. 서비스 목표**

1. 주변 사용자의 현재 음악 상태를 실시간으로 확인할 수 있게 한다.  
2. 정확한 위치나 얼굴을 공개하지 않고 주변 사용자의 존재를 표현한다.  
3. 음악 취향 유사도를 기준으로 관심 있는 사용자를 탐색할 수 있게 한다.  
4. 맞팔 이전의 자유 입력 메시지를 제한해 무작위 메시지와 스팸을 줄인다.  
5. 다수 사용자가 참여하는 실시간 음악 공간을 제공한다.  
6. 앱이 백그라운드로 이동한 뒤에도 사용자가 시작한 주변 공유를 유지한다.  
7. 인터넷 연결이 불안정한 환경에서도 제한적인 음악 카드 교환을 지원한다.  
8. 단일 WebSocket 연결에서 개인 통신, 그룹 통신, 지역 브로드캐스트를 처리한다.

## **3\. 핵심 설계 원칙**

### **3.1 정확한 위치를 제공하지 않는다**

Melody Bubble은 위치 추적 서비스가 아니라 근거리 음악 소셜 서비스다.

다른 사용자에게 다음 정보는 제공하지 않는다.

* GPS 좌표  
* 정확한 거리  
* 정확한 방향  
* 이동 경로  
* 건물 층  
* 좌석 위치

서버는 설정된 탐색 범위 안의 사용자만 검색하고, 클라이언트에는 버블 표시용 가공 데이터만 전달한다.

탐색 거리는 고정된 서비스 정책값으로 관리한다.

예시 탐색 반경:

100m

200m

300m

정확한 값은 실제 캠퍼스 환경에서 사용자 밀도와 GPS 오차를 측정한 뒤 결정한다.

### **3.2 사용자가 직접 주변 공유를 시작한다**

주변 공유는 자동으로 실행하지 않는다.

사용자가 앱 화면에서 `주변 공유 시작` 버튼을 누르면 Android Location Foreground Service가 시작된다.

사용자가 공유 시작

→ Foreground Service 실행

→ 지속 알림 표시

→ 위치 및 음악 상태 전송

→ 앱이 백그라운드로 이동해도 공유 유지

Foreground Service 실행 중에는 다음과 같은 알림을 표시한다.

Melody Bubble 주변 공유 중

현재 음악과 대략적인 위치가 주변 사용자에게 공개됩니다.

\[공유 중지\]

사용자는 앱 또는 알림에서 언제든 공유를 종료할 수 있다.

### **3.3 WebSocket을 핵심 통신 계층으로 사용한다**

실시간성이 필요한 기능은 STOMP over WebSocket을 사용한다.

* 위치 업데이트  
* Presence heartbeat  
* 주변 사용자 입장·이탈  
* 현재 음악 상태  
* 음악 리액션  
* 맞팔 알림  
* 채팅  
* 입력 중 상태  
* 메시지 읽음 상태  
* 지역 음악 트렌드  
* 음악 라운지  
* 행사 실시간 반응  
* 실시간 투표

회원가입, 설정 변경, 과거 기록 조회 등 요청·응답 중심 기능은 REST API를 사용한다.

### **3.4 개인 정보와 집계 정보를 분리한다**

개별 사용자의 주변 정보는 공개 Topic으로 전송하지 않는다.

사용자마다 다음 조건이 다르기 때문이다.

* 차단 관계  
* 음악 공개 범위  
* 주변 노출 여부  
* 맞팔 여부  
* 취향 유사도 공개 조건

주변 사용자 정보는 서버가 수신자별로 계산한 후 개인 Queue로 전달한다.

지역 Topic에는 개별 사용자를 식별할 수 없는 집계 정보만 전송한다.

### **3.5 자동 음악 감지는 보조 기능으로 제공한다**

현재 음악 자동 감지는 Android MediaSession을 이용한다.

다만 음악 앱별 구현 차이가 있으므로 모든 앱에서 동일한 결과를 보장하지 않는다.

자동 감지가 불가능하면 다음 대체 경로를 제공한다.

* 곡 직접 검색  
* 최근 선택 곡  
* 대표곡 설정  
* 제목·아티스트 직접 입력  
* 외부 음악 앱 링크 등록

## **4\. 주요 사용자 시나리오**

## **4.1 주변 공유 시작**

1. 사용자가 로그인한다.  
2. `주변 공유 시작`을 누른다.  
3. 위치 권한을 확인한다.  
4. 음악 자동 감지를 사용하는 경우 알림 접근 권한을 확인한다.  
5. Location Foreground Service를 시작한다.  
6. WebSocket 연결을 생성한다.  
7. STOMP CONNECT를 수행한다.  
8. 개인 Queue와 필요한 Topic을 구독한다.  
9. 위치, 음악 상태, heartbeat를 서버에 전송한다.  
10. 서버가 주변 사용자를 검색한다.  
11. 사용자의 버블맵에 주변 사용자가 표시된다.

## **4.2 백그라운드 공유**

1. 사용자가 주변 공유를 시작한다.  
2. 홈 화면으로 이동하거나 다른 앱을 사용한다.  
3. Foreground Service가 WebSocket 연결과 위치 업데이트를 유지한다.  
4. 음악 상태가 변경되면 서버에 실시간으로 전송한다.  
5. 주변 사용자 목록 변화는 개인 Queue로 수신한다.  
6. 사용자가 앱으로 돌아오면 최신 버블맵 상태를 표시한다.

## **4.3 주변 사용자와 음악으로 연결**

1. 사용자가 주변 버블을 선택한다.  
2. 상대방이 공개한 음악 정보를 확인한다.  
3. 음악 취향 유사도와 공통 장르를 확인한다.  
4. 음악 리액션 또는 팔로우를 보낸다.  
5. 상대방도 팔로우하면 맞팔이 성립한다.  
6. 맞팔 알림을 WebSocket으로 수신한다.  
7. 1대1 채팅방이 활성화된다.

## **4.4 실시간 음악 라운지**

1. 사용자가 현재 지역이나 행사에 연결된 음악 라운지에 입장한다.  
2. 라운지 Topic을 구독한다.  
3. 참여자 수와 현재 음악 분위기를 확인한다.  
4. 추천곡 카드 또는 정해진 리액션을 전송한다.  
5. 라운지의 모든 참여자가 실시간으로 결과를 수신한다.  
6. 사용자가 장르·분위기·추천곡 투표에 참여한다.  
7. 투표 결과가 모든 참여자 화면에 반영된다.

## **4.5 오프라인 음악 카드 교환**

1. 두 사용자가 근거리 음악 카드 교환 기능을 실행한다.  
2. Nearby Connections가 교환 모드를 실행 중인 기기를 탐색한다.  
3. 사용자가 연결할 상대를 선택한다.  
4. 양쪽 기기에 같은 인증 코드를 표시한다.  
5. 양쪽 사용자가 연결을 승인한다.  
6. 음악 카드와 멜로디 별칭을 교환한다.  
7. Room Database에 교환 기록을 저장한다.  
8. 연결을 종료한다.  
9. 인터넷이 복구되면 서버에 교환 기록을 동기화한다.

## **5\. 핵심 기능**

## **5.1 주변 공유 세션**

사용자가 주변 공유를 시작하면 Foreground Service가 다음 기능을 수행한다.

* 위치 업데이트  
* 현재 음악 상태 확인  
* WebSocket 연결 관리  
* STOMP 구독 관리  
* Presence heartbeat  
* 네트워크 재연결  
* 지속 알림 표시  
* 공유 종료 처리

### **권장 업데이트 주기**

위치 업데이트:

15\~30초 간격

또는 일정 거리 이상 이동 시

Presence heartbeat:

30초 간격

Presence TTL:

90초

위치 업데이트는 시간 조건과 이동 거리 조건을 조합한다.

### **공유 세션 종료 조건**

* 사용자가 공유 중지  
* 위치 권한 해제  
* 사용자가 Foreground Service 중지  
* 앱 강제 종료  
* 기기 종료  
* Presence TTL 만료  
* 서버가 계정 이용을 제한

### **자동 재시작 제한**

다음 상황에서는 자동 재개를 보장하지 않는다.

* 앱 강제 종료  
* 기기 재부팅  
* 권한 해제  
* 시스템 서비스 제한

사용자는 앱을 다시 열어 주변 공유를 시작해야 한다.

## **5.2 주변 사용자 버블맵**

주변 사용자를 익명 버블로 시각화한다.

### **표시 요소**

* 프로필 색상  
* 음악 파형  
* 재생 여부  
* 장르 아이콘  
* 분위기 아이콘  
* 멜로디 별칭  
* 취향 유사도  
* 신규 사용자 효과

### **버블 데이터 예시**

{

  "temporaryUserId": "temp-28f1",

  "displayPosition": {

    "x": 0.68,

    "y": 0.31

  },

  "matchScore": 82,

  "visualType": "WAVE",

  "isPlaying": true,

  "primaryGenre": "INDIE",

  "mood": "CALM"

}

`displayPosition`은 화면 배치용 값이며 실제 방향이나 좌표를 나타내지 않는다.

### **버블 배치 정책**

* Jetpack Compose Canvas 사용  
* 버블 중첩 최소화  
* 유사도 높은 사용자 우선 표시  
* 사용자 수가 많을 경우 일부 사용자만 표시  
* 이동 시 애니메이션 적용  
* 동일 사용자의 급격한 위치 변경 방지  
* 오래된 상태는 TTL 만료 후 제거

## **5.4 음악 취향 프로필**

### **사용자 입력**

* 선호 장르  
* 선호 아티스트  
* 대표곡  
* 분위기 태그  
* 최근 앱 내 선택 곡  
* 음악 리액션 기록

### **유사도 계산**

장르 유사도           35%

아티스트 유사도       30%

최근 선택 곡 유사도    20%

분위기 태그 유사도     15%

matchScore \=

    0.35 × genreSimilarity

  \+ 0.30 × artistSimilarity

  \+ 0.20 × recentTrackSimilarity

  \+ 0.15 × moodSimilarity

### **계산 방식**

* 집합 기반 Jaccard 유사도  
* 빈도 벡터 코사인 유사도  
* 서버 내부 가중 합산  
* 결과 범위 0\~100

MVP에서는 LLM을 추천 핵심 계산에 사용하지 않는다.

## **5.5 음악 리액션**

맞팔 이전에는 자유 입력 텍스트를 허용하지 않는다.

### **리액션 종류**

* 이 노래 좋아해요  
* 이 아티스트 저도 좋아해요  
* 비슷한 음악을 듣고 있어요  
* 대표곡이 마음에 들어요  
* 플레이리스트에 추가했어요  
* 지금 분위기와 잘 어울려요

### **통신 흐름**

Android

→ /app/reaction/send

→ 서버 권한 및 rate limit 검증

→ DB 저장

→ 수신자 개인 Queue 전송

### **제한 정책**

* 동일 상대 반복 전송 제한

* 비공개 사용자에게 전송 불가  
* 계정 정지 사용자는 전송 불가  
* 서버에서 전송 빈도 제한  
* 사용자가 리액션 수신 비활성화 가능

## **5.9 Music Pulse**

Music Pulse는 특정 지역에서 발생하는 음악 반응을 짧은 애니메이션으로 공유하는 기능이다.

### **Pulse 예시**

* 곡 좋아요  
* 같은 곡 청취  
* 추천곡 급상승  
* 장르 반응 증가  
* 행사 분위기 변화

### **이벤트 예시**

  }

}

### **개인정보 처리**

* 사용자 ID 미포함

* 정확한 위치 미포함  
* 장기간 저장하지 않음  
* 개인 행동과 연결하지 않음

### **핵심 온라인 흐름**

사용자가 주변 공유 시작

→ Foreground Service 실행

→ STOMP WebSocket 연결

→ 개인 Queue 및 지역 Topic 구독

→ 위치·음악·heartbeat 전송

→ Spring Boot 권한 검증

→ PostGIS 주변 사용자 검색

→ 개인 Queue로 주변 사용자 Delta 전달

→ RabbitMQ를 통해 지역·라운지 Topic 브로드캐스트

→ Compose UI 실시간 반영

### **핵심 오프라인 흐름**

사용자가 근거리 교환 시작

→ Nearby Connections 기기 탐색

→ 상대 선택

→ 인증 코드 확인

→ 음악 카드 교환

→ Room 저장

→ 인터넷 복구 후 REST 동기화

Melody Bubble은 서버 기반 주변 사용자 검색과 STOMP WebSocket 기반 실시간 통신을 중심으로 구현한다. 개인화된 주변 사용자 정보와 채팅은 사용자별 Queue로 전달하고, 음악 라운지·지역 트렌드·행사 반응은 RabbitMQ Topic을 통해 다수 사용자에게 동시에 전달한다. 백그라운드 공유는 사용자가 직접 시작한 Android Foreground Service가 담당하며, 오프라인 교환은 Nearby Connections가 담당한다.
